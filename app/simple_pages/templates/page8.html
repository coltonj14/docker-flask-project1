SOLID Design

SOLID in the Calculator

The SOLID principles can be seen throughout the calculator design. The first principle, Single-responsibility principle can be seen directly in the operations classes where the only job of each class is to return the result of an operation between two values.

The next principle, open-close principle, can be seen in the fact that in the calculation class each subclass has its own get_result() method. This is better so that there is not one big method with if-else statments to figure out which operation we are trying to perform.
The Liskov substitution principle is satisfied through the subclasses of Calculation as they perform the operations on the values and return the final sum.
The interface segregation principle is satisfied throughout the design of the calculator as there are no methods that are not applicable to the problem at hand. Every method in the classes is usable whether it be a subclass or parent class.
The final principle of Dependency Inversion can also be seen in the calculator implementation. This principle is most evident with the Calculator class depending on the abstract methods in the Calculation class. This helps to satisfy this principle as it is really the only place where it is applicable.

Design method - Factory Method

The Factory method of design can be seen in this calculator project. The factory method of design has a purpose of returning back an instance of the class you are referring to.

Here we can see we are returning the instance of Multiplication class which is renamed to Mult.
