OOP

OOP programming terms glossary

  • Instantiation

    • Instantiation is essentially the creation of an object in OOP. When instantiating we are creating an "Instance" of that particular class. For example creating an instance of the Calculator class allows you to work with a Calculator object.

  • Object

    • An object is an instance of a class.

  • Class

    • A class is a blueprint for an object. It contains all the necessary data and methods to build and use an object.

  • Namespace

    • A namespace is a collection of the currently defined names as well as the information about the object that it references.

  • Constructor

    • A constructor is a special method in a class that has the same name as the class. This method is executed when the class is instantiated, often to set up the object. This setup can be for object properties or instance variables.

  • Fixture

    • Fixtures help in the arranging and some of the act phase in AAA testing. It helps to provide the clear steps and data that will be used in the test.

  • Type hint

    • Since some languages are dynamically typed type hinting is way to make it more static. For example, python is a dynamically typed language which essentially means that you do not have to declare the variable type. Using type hints in python can help to make tests very clear.

  • Type cast

    • Type casting is used when you want to change a variable to another type. For example, in python if you want to represent the string "123" as an integer you can type cast it like this int("123")

  • Unit test

    • A unit test is a way for developer to test individual components of a software or application. This is to make sure that the code is working as intended.

  • Static

    • Static refers to a part of the class that belongs to the class whether it be a property or method.

  • Instance method

    • Instance method are the typical methods that you would see in a class that belong to the object instances of the class. Often times in an instance method will include self in the parameters to refer to the instance of the object itself.

  • Instance property

    • A property that belongs to a particular object. Often times these are instantiated in the constructor.

  • Static method

    • Adding static identifier to a method in a class means that the method belongs to the class and not the object.

  • Static property

    • When a variable is declared inside the class but not in a method it is referred to as a static variable or property. This property can be used directly form the class but not directly by an instance of the class.

  • Encapsulation

    • One of the four main principles of OOP. Encapsulation is the idea that data and methods are "encapsulated" within a class. This helps to restrict access to data or methods within a class. An example of encapsulation is having private static variables that are accessable only by a getter method within the class.

  • Inheritance

    • One of the four main principles of OOP. Inheritance is the property that a class can inherit properties from another class. This helps to provide reusability of code as well as keep the code organized.

  • Polymorphism

    • One of the four main principles of OOP. Polymorphism boils down to being able to have the same signature with different method actions. For instance, in the operation class(which are subclasses of the Calculation class) have their own get_result() method with a slightly different implementation.

  • Abstraction

    • One of the four main principles of OOP. Abstraction allows for classes and method to essentially act as a blueprint for the class. Abstract classes or methods provide the structure but not the implementation.

Overview of SOLID

  • S - Single-responsibility principle -> States that a class should only have one job.
  • O - Open-close principle -> Objects should be open for extension but closed for modification.
  • L - Liskov substitution principle -> Every subclass should be substitutable for their parent class.
  • I - Interface segregation principle -> Should not be forced to depend on methods they don't use.
  • D - Dependency inversion principle -> Entities must depend on abstraction.

Design Patterns

Design patterns are solutions to existing software design problems that provide a clear pattern for implementation. There are three main design patterns and they are creational, behavioral, and structural.
